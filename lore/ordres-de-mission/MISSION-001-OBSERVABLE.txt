BRIEFING OPÉRATIONNEL

Au cœur du Terminal de Combat pulsent des observables — des flux de valeurs
auxquels les composants s'abonnent pour réagir en temps réel. Cette
infrastructure réactive n'appartient pas à Core Nexus : elle est
universelle, réutilisable, publiable.

@efficimo/observable est un système de programmation réactive minimaliste,
sans aucune dépendance obligatoire. Il expose des primitives composables :
Observable (événements), ObservableValue (état réactif avec historique),
DerivedObservableValue (valeur dérivée bidirectionnelle). Chaque abonnement
retourne une Subscription pour se désabonner proprement.

La lib existe déjà dans core-nexus sous src/observable/ — cette mission
l'extrait, la nettoie, la teste et la publie sur npm.

API PUBLIQUE

  PRIMITIVES CORE (zéro dépendance)
  ──────────────────────────────────

  // Émetteur d'événements
  class Observable<T> {
    subscribe(fn: (value: T) => void): Subscription
    next(value: T): void
  }

  // État réactif — émet la valeur courante immédiatement à l'abonnement
  class ObservableValue<T> extends Observable<T> {
    constructor(initialValue: T)
    getValue(): T
    next(value: T | SetFunction<T>): Promise<void>   // déduplique via isEqual()
    subscribe(fn: (value: T) => void): Subscription  // reçoit la valeur courante d'emblée
  }

  // Valeur dérivée bidirectionnelle — synchronise deux ObservableValue
  class DerivedObservableValue<T, U> extends ObservableValue<T> {
    constructor(
      source: ObservableValueInterface<U>,
      from: (u: U) => T,         // source → dérivé
      to:   (t: T) => U,         // dérivé → source (write-back)
      defaultValue?: T
    )
  }

  // Abonnement — retourné par subscribe()
  class Subscription implements SubscriptionInterface {
    unsubscribe(): void
  }

  // Utilities internes, exportées pour les consommateurs
  function isEqual(a: unknown, b: unknown): boolean
  type SetFunction<T> = (current: T) => T | Promise<T>

  ADAPTATEUR ZOD (peerDependency: zod ≥ 3)
  ──────────────────────────────────────────
  Importé via '@efficimo/observable/zod'

  // Observe une valeur string | null, l'interprète comme JSON via un schéma Zod
  class JsonSerializeObservableValue<T> extends DerivedObservableValue<T | null, string | null> {
    constructor(
      source: ObservableValueInterface<string | null>,
      schema: ZodType<T>,
      defaultValue?: T
    )
  }

  HOOK REACT (peerDependency: react ≥ 18)
  ─────────────────────────────────────────
  Importé via '@efficimo/observable/react'

  // Lie un ObservableValue à l'état React — retourne [valeur, setter]
  function useObservableValueState<T>(
    observable: ObservableValueInterface<T>
  ): [T, (value: T | SetFunction<T>) => Promise<void>]

  INTERFACES (pour les consommateurs TypeScript)
  ───────────────────────────────────────────────
  interface SubscriptionInterface { unsubscribe(): void }
  interface ObservableInterface<T> { subscribe(...): Subscription; next(value: T): void }
  interface ObservableValueInterface<T> extends ObservableInterface<T> { getValue(): T }

     - Remplacer @/observable/JsonSerializeObservableValue → @efficimo/observable/zod

  6. VÉRIFIER que core-nexus compile après migration (pnpm turbo build)

RAPPORT D'ANALYSE

  ★ APPRÉCIATION
  Code déjà mature — la migration est principalement mécanique. Le design
  est solide : Observable sans état, ObservableValue avec déduplication,
  DerivedObservableValue bidirectionnel. La séparation en trois points
  d'entrée (core / react / zod) permet d'utiliser @efficimo/observable
  dans des contextes non-React (Node.js, tests, Plexus côté Worker) sans
  embarquer React inutilement.

  ⚠ POINTS DE VIGILANCE TECHNIQUES

  OPTIONAL PEER DEPS
    peerDependenciesMeta.optional = true indique au gestionnaire de paquets
    que zod et react ne sont pas requis. Sans cette déclaration, pnpm et
    npm afficheront des avertissements chez les consommateurs qui n'utilisent
    pas l'adaptateur Zod ou le hook React.

  IMPORTS CONDITIONNELS
    src/zod.ts et src/react.ts ne s'importent que si le consommateur les
    utilise explicitement. Les bundlers modernes (Vite, esbuild) ne
    transpileront pas ces fichiers si l'entrée "./zod" n'est jamais
    importée. Pas de tree-shaking à gérer manuellement.

  TESTS SANS DOM
    Les tests du core (Observable, ObservableValue, isEqual) ne nécessitent
    pas jsdom — Vitest peut tourner en environnement node pur. Le test du
    hook React nécessite @testing-library/react avec jsdom.

  DÉDUPLICATION isEqual
    ObservableValue.next() ne notifie pas les abonnés si la nouvelle valeur
    est égale à l'ancienne (via isEqual). Ce comportement est intentionnel
    mais peut surprendre : un objet re-créé avec les mêmes valeurs ne
    déclenchera pas de mise à jour.

  ✦ RECOMMANDATION
    Commencer par écrire les tests avant de migrer — ils servent de filet
    de sécurité pour vérifier que le comportement est intact après le
    déplacement. Ordre suggéré : tests isEqual → Observable → ObservableValue
    → Derived → migration des imports core-nexus → vérification build.
