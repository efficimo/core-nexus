ACTE II — C'EST QUOI EXACTEMENT ?
───────────────────────────────────

Les Ordres de Mission sont des micro-projets d'apprentissage autonomes.
Pas des exercices de synthèse inventés. Pas des kata déconnectés du réel.

Chaque mission correspond à une brique réelle de notre stack :
un package npm qu'on publie, une feature qu'on intègre, un concept
qu'on pousse jusqu'au bout — dans notre environnement, avec nos outils.

  ┌──────────────────────────────────────────────────────────┐
  │ Ce ne sont pas des tutoriels à suivre.                   │
  │ Ce sont des specs à implémenter.                         │
  │                                                          │
  │ La mission dit "quoi" et "pourquoi".                     │
  │ C'est vous qui décidez "comment".                        │
  └──────────────────────────────────────────────────────────┘

OBJECTIFS GLOBAUX

  → Comprendre comment fonctionne notre stack de l'intérieur
  → Contribuer à des packages open source réels (@efficimo/*)
  → Développer un raisonnement d'architecte, pas juste d'exécutant
  → Apprendre à lire une spec technique et à la traduire en code

ACTE VII — LE GRAND FINALE : PLEXUS
────────────────────────────────────

[Garder pour la fin. Laisser le silence s'installer avant de parler.]

"Et maintenant, la raison pour laquelle tout ce qu'on vient de voir existe."

  MISSION-008  🕸️  plexus — Réseau Mesh Fantôme
  ────────────────────────────────────────────────
  Un réseau P2P entre navigateurs. Sans serveur. Sans WebSocket.
  Chaque onglet est un nœud. Chaque nœud connaît ses voisins.
  Ensemble, ils savent tout.

  Techniquement : routage distance-vector entre onglets Chrome,
  via WebRTC + BroadcastChannel. La même logique que les routeurs
  Internet depuis les années 70 — appliquée à des tabs de navigateur.

  Un message d'Alice vers Bob emprunte le chemin le plus court
  calculé dynamiquement par les tables de routage de chaque nœud.

  C'est la mission la plus complexe. C'est aussi celle qui illustre
  le mieux la philosophie du projet : "pas de backend" ne veut pas
  dire "pas de réseau". Ça veut dire "le réseau, c'est vous."

  Dépendances : observable + BroadcastChannel (016) + Signal (032)
  Niveau : ★★★★☆ — Champion

  MISSION-009  🌌  plexus-ui — Interface Constellation
  ────────────────────────────────────────────────────────
  La visualisation SVG en temps réel du réseau Plexus.
  Chaque nœud actif apparaît. Chaque connexion est tracée.
  On voit le mesh vivre.
  Niveau : ★★★☆☆

  ┌────────────────────────────────────────────────────────────────┐
  │ POURQUOI PLEXUS EN DERNIER ?                                   │
  │                                                                │
  │ Parce que pour comprendre Plexus, il faut avoir compris        │
  │ observable (réactivité), BroadcastChannel (communication       │
  │ inter-onglets), et WebRTC (P2P navigateur).                    │
  │                                                                │
  │ Et parce que c'est le projet qui prouve que notre stack        │
  │ peut aller aussi loin qu'on veut — sans jamais quitter         │
  │ le navigateur.                                                 │
  └────────────────────────────────────────────────────────────────┘

ACTE VIII — RÈGLES DU JEU
───────────────────────────

  1. UNE MISSION À LA FOIS
     On ne jongle pas entre 3 missions. On finit ce qu'on commence.

  2. LIRE LA SPEC AVANT DE CODER
     Chaque mission contient tout : objectifs, structure, interfaces,
     pièges. 10 minutes de lecture = 2 heures de débogage évitées.

  3. LES LIVRABLES SONT DES CHECKLISTS
     Cocher = c'est fait ET ça compile ET c'est testé.
     Pas de "ça marche chez moi" sans tests.

  4. ON POSE DES QUESTIONS, ON NE SE NOIE PAS
     Si tu es bloqué 45min sur le même point, tu demandes.
     Chercher longtemps = apprendre. Tourner en rond = perte de temps.

  5. LE CODE EST OPEN SOURCE
     Ce qu'on produit dans les packages @efficimo/* est public sur npm.
     On écrit pour des inconnus. On documente pour soi-même dans 6 mois.
